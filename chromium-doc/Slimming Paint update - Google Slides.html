Old vs new

Before:

  • Composite based on LayoutObject subtrees
  • Invalidate based on bounding rects

After:

  • Composite based on Display List chunks
  • Invalidate based on LayoutObjects

The compositor maps this to a rect in of its composited layers, and re-rasters that rect.

26 of 42

Why Slimming Paint? Recap.

  • Blink sometimes paints incorrectly. This is not fixable in the current system.

  • Uncomfortably exciting amount of complexity in Blink compositing.

27 of 42

Some more reasons

  • GPU Rasterization (aka Ganesh) is in conflict with compositing
  • Compositor-driven effects cause inefficient compositing
  • Future-looking:
    • Extremely fast, targeted paint becomes possible. Think changing background of a single element
    • Output of paint becomes independent of Blink yet retains semantics.

28 of 42

Implementation highlights

  • Invalidate individual LayoutObjects
  • Generate a Display List. Mutate it on each frame, and merge into the previous one
  • Build auxiliary data structures for the compositor
  • Implement composited layerization
  • Raster quickly

29 of 42

Invalidation

  • Invalidate individual LayoutObjects rather than paint rectangles.
  • Store invalidation bits on such objects (these already exist).
  • Challenge: naive implementation has lots of under-invalidation.

30 of 42

Painting

  • Walk the LayoutObject tree in paint order (i.e. multiple passes over each object).
    • For invalidated LayoutObjects, output display items keyed by <LayoutObject*, paint phase>.
    • For uninvalidated ones, output caching placeholders with the same key.
  • Do special work for non-LayoutObjects like scrollbars, plugins, etc.

31 of 42

DisplayItem internal examples

  • DrawingDisplayItem
    • Wrapper for SkPicture
  • BeginClipDisplayItem / EndClipDisplayItem
    • Stores square and rounded clips for bracketed content
  • BeginTransform3DDisplayItem / EndTransform3DDisplayItem
    • Stores a 4x4 transform matrix + 3D transform origin
  • BeginFixedPositionContainerDisplayItem / EndFixedPositionContainerDisplayItem
    • Placeholder to indicate fixed-position. Positioned relative to containing transform node.

Most display item types map directly to the Skia API.

32 of 42

Merge into the previous display list

  • Walk the partial paint list, replacing CachedDisplayItems with ones from the previous frame.
  • Optimized to reduce size of DisplayItem hash maps and for unchanged segments.

33 of 42

Auxilliary data structures

Recall other uses of composited layers:

  • Compositor-driven animations
  • Compositor-driven input events and scrolling

We also need data structures to make the compositors job easier:

  • Computing visible content rects, content clipping, opacity/filters, overlap testing, image detection, and raster culling

Property trees solve some of the above use cases

34 of 42

Property trees

  • Transform
    • All nodes in the document with 3D transforms
  • Clipping
    • All nodes with overflow: anything
  • Scroll
    • Special case of transform
  • Opacity/filter
    • All nodes with such style

Property tree implementation

  • Can be computed from the display list
    • Mutated based on partial display list updates
  • Can also be computed from the existing cc::Layer tree
    • More on this later in phase 1

36 of 42

Compositor layerization

  • Split display list into chunks at compositing reason id boundaries
  • Composite composited chunks that are from the same id
  • Composite chunks that overlap existing ones, squashing multiple together as appropriate
